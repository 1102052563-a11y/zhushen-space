## 五、★「对不上」三处线上 bug 修复（本次续接会话）

### 5a. ★HP/EP 血条与 `<状态结算>` 对不上（91≠92 / 49≠57）
- **根因**：`playerMaxHp/Ep` 硬传 `realMult=1`，血条/属性面板/战斗却用 `realAttrMult(阶位,等级)`（四阶起×5）→ 四阶+主角「钳制上限」偏低、把正文当前值钳下去。
- **修**：`src/systems/playerVitals.ts` `playerMaxHp/Ep` 的 `1` → `realAttrMult(prof.tier, prof.level)`，两处口径对齐。
- **测**：`src/systems/playerVitalsRealMult.test.ts`（四阶×5 · 三阶不放大）。

### 5b. ★世界之源「正文总计6.3% / 侧栏4」对不上 + 每世界归零
- **根因**：世界之源只由主角演化阶段读隐藏块 `<世界之源>` 绝对总量落库，与主叙事（另一次 AI 调用）可见的「当前总计」基数看法不一致 → 分叉。
- **修（双管）**：
  - 代码 `src/systems/statusCommands.ts`：主角演化阶段拿到主叙事 → **以玩家可见的「世界之源…总计 X%」为准**覆盖块值（只认 累计/总计/合计 关键词、避开「获得3.5%」单回合增量）；**人在乐园（轮回乐园/专属房间/主神空间）强制归零**。
  - 代码 `src/systems/playerVitals.ts` `reconcileHomeWorld`：回归乐园每回合确定性把 worldSource 归 0（不靠 AI 记得发 `=0`）。
  - 提示词 `src/promptRules.ts` `WORLDSOURCE_RULE`：隐藏块总量必须＝正文给玩家看的「当前总计」（照抄）。
  - `src/systems/acuTableSpec.ts`：`world_source` DDL `INTEGER`→`REAL`（百分比含小数）。
- **测**：`src/systems/statusCommands.worldSource.test.ts`（可见总量优先 · 单回合增量不误判 · 人在乐园强制归零 8 例）。

### 5c. ★主角血条读成了 NPC 血量（14≠96 / 1≠61）
- **根因**：`<状态结算>` 逐行一个角色；旧 `grabLast` 取「最后一个非他人」的当前HP，刚解救未建档的「女性村民」不被 `OTHER_RE` 认识 → 其 14/20 被当成主角；EP 更因主语识别不到而串行。
- **修**：`src/App.tsx` `applyNarrativeVitals` 优先认「以 主角/你/我 开头的整行」，从该行**同时取 HP+EP**（与 NPC 是否建档无关），取不到才回退 `grabLast`。
- **验**：独立脚本按 bug 原文 + 6 边界全 PASS（不可导入单测·函数内嵌 App.tsx）。

### 5d. ★结算货币「面板 7000 / 入账 4000」对不上
- **根因**：世界结算 `<世界结算>` 面板只展示，之后 `<state>` 用 `乐园币 += {总货币}` 真入账。AI 面板写"获得货币: 7000"、`<state>` 却发 `乐园币 += 4000`——面板与入账指令自相矛盾，store 忠实执行 4000（同 worldSource：可见值≠指令值）。
- **修（忠于面板·双管）**：
  - 代码 [stateApply.ts `reconcileSettlementCurrency`](zhushen-space/zhushen-space/src/systems/stateApply.ts:355)：结算回合解析【最终清算】"获得货币: N 乐园币/灵魂钱币"，把 `<state>` 里 `乐园币/灵魂钱币 += M` 校正成 N（含同类去重防"统计+发放"双入账）；接进 `applyStateUpdates`。
  - 提示词 [WORLD_SETTLEMENT_RULE](zhushen-space/zhushen-space/src/promptRules.ts)：`乐园币 += N` 的 N 必须＝面板"获得货币"那个数。
- **测**：`src/systems/settlementCurrency.test.ts`（9 例：覆盖 bug 原文 / 灵魂钱币 / 多条去重 / 类型不匹配不误改 / 千分位 / = 不碰）。

## 十三、Bug 修复：最近删除刷新丢失 + 重生成/回退闪主菜单
**① 最近删除刷新后消失**：`itemStore` 的 `partialize` 只给 `items` 剥掉大体积 `image`（图在 IndexedDB），**没给 `recentlyDeleted` 剥**——删掉带生图的装备后，回收站条目连 base64 图一起写进 `drpg-items`，撑爆 localStorage 配额 → persist 静默失败 → 这次删除没落盘 → 刷新后回收站空了。（实测：无图时普通 reload 能存活，证明纯持久化没坏、就是体积问题。）修：partialize 给 `recentlyDeleted` 也剥 `image` + 条目封顶 100。

**② 重生成/回退总闪一下主菜单**：⟳重新生成 / ↩回退 / ♻重算变量 / 读档都靠 `loadSlot→reload` 恢复（gameStore 无 rehydrate）。reload 后 `started` 初值是 `false` → 先渲染 StartScreen → 挂载的 **async** effect 读 sessionStorage 的 `PENDING_*` 标志后才 `setStarted(true)` → 闪一下主界面再切回。修：`started` 用惰性初始化**同步**读 `PENDING_STARTED/REGEN/REVAR` → 首帧就进游戏界面。

## 二十二、Bug 修复：正文提到已有武器→重复生成（月影残心 ×3·含带【】那把）
**现象**：正文一提到已有装备，背包立马多出两把同名（截图：`月影残心`×2 + `【月影残心】`×1，都金色）。**根因＝两套名称归一器不一致 + 判重太严**：① `stateParser.normName`（`findIdenticalItem` 创建闸门用）**不剥【】括号**，而 `itemStore.dedupeByName` 的 norm 剥——于是「【月影残心】」≠「月影残心」判成两物；② `findIdenticalItem` 要 **name+grade+effect+combatStat 全等**才判重，但 AI 重提同一件武器时**描述/效果每次都变**（"金色·顶级单手长刀" vs "金色品质武器"）→ effect 不等→漏判成新物。双重漏网 → 重复生成。

| 文件 | 作用 | 状态 |
|---|---|---|
| `src/systems/stateParser.ts` `normName` | 归一口径对齐 `dedupeByName`：加剥 `【】〔〕「」『』〈〉（）()` 装饰括号 + `的之` 结构助词 → 「【月影残心】」归一后＝「月影残心」 | 改动 |
| `src/systems/stateParser.ts` `findIdenticalItem` | 装备判重放宽为 **同名 + 品级宽松吻合**（任一空/互相包含·治「金色 vs 金色·顶级」判同、「金色 vs 红色·升级」放行）；**去掉 effect/combatStat 硬性条件**（AI 重提描述必变）。仍是 `skip` 拦截（不新建·**不删物**，绕开「按名合并→丢装备」老坑） | 改动 |
| `src/systems/ledger/itemLedger.test.ts` | 更新 1（同名不同效果→现判重只一件）+ 新增 3（【】括号判重 / 品级宽松 金色vs金色·顶级 / 不同品级 金色vs红色放行） | 改动 |

> 验证：itemLedger 30 单测绿（去重四场景：同批/跨批/括号/品级宽松·升级放行）· tsc 干净 · **818 绿**（全套件·含外部并行新增测试）· build 过。设计权衡：**创建闸门 skip（非事后合并）**——只是"不新建重复"，绝不删已有装备，故不会重蹈 `dedupeByName` 注释里记的「按名合并悄悄吞掉一件装备」老坑。⚠ 一个偶发：全套件里外部未跟踪的 `regexEngine.test.ts`（正则视图作用域 WIP）偶尔因跨文件排序污染报 `$1` 失败——**单独跑 10/10 绿、与本改动无关**（regexEngine 只 import 一个 type·不碰我改的文件），属外部 WIP 的 flaky，本次已复现过一次绿。
