# 存档功能集成指导

参考 **fanren-remake** 的存档系统，给 **zhushen-space** 加「多存档槽（创建 / 读取 / 改名 / 删除 / 导出 / 导入）」。

---

## 0. 一句话

一个「存档」= 某一时刻**全部游戏状态 + 全部对话历史**的快照。多个存档并存，可随时切换/读取，可导出成 `.json` 文件备份、再导入。

---

## 1. fanren-remake 怎么做的（源码定位）

存档系统基于 **IndexedDB（Dexie 封装，`assets/db-X5K5aqCB.js`）**，不是 localStorage。核心对象仓（object stores）：

| store | 作用 |
|---|---|
| `archiveMetas` | 每个存档的**元信息**：id、name、description、createdAt、（更新时间、回合数、封面图 id 等）|
| `archives` | 每个存档的**游戏状态**（角色/世界/物品/变量…）|
| `archiveChatMessages` | 每个存档的**对话/正文消息**（按 archiveId 关联）|
| `archiveLifeSnapshotPayloads` | 生平/深度记忆快照负载 |
| `settings` | 全局设置 |

- `activeArchiveId`：当前激活的存档 id（`index-Zjr69bst.js` / `uiStore`）。
- 操作：`createArchive` / `switchArchive`（读取/切换）/ `renameArchive` / `deleteArchive` / `downloadArchive`（导出为文件，`service-xRZ1P_7x.js`）/ `importArchive`。
- **localStorage → IndexedDB 迁移**（`index-Zjr69bst.js` 的 `vn()`）：老版本存在 localStorage 的存档会被迁进 IndexedDB。
- 为什么用 IndexedDB 而非 localStorage：对话历史 + 多存档体积大，localStorage 只有 ~5MB，IndexedDB 容量大得多。

## 2. zhushen-space 现状

| 维度 | fanren-remake | zhushen-space 现状 | 差距 |
|---|---|---|---|
| 持久化 | IndexedDB 多存档 | Zustand `persist` 到 **localStorage**，**单一隐式存档** | 无多槽 |
| 涉及的 store | 统一打包 | 9 个 `drpg-*` key：`drpg-save`(gameStore)/`drpg-settings`/`drpg-items`/`drpg-player-evo`/`drpg-npc`/`drpg-npc-evo`/`drpg-characters`/`drpg-memory`/`drpg-misc` | 需统一快照 |
| 对话历史 | `archiveChatMessages` 持久化 | `App.tsx` 的 `messages` 是 **useState，内存态、刷新即丢** | **必须先持久化** |
| 入口 | 存档界面 | StartScreen「读取存档」按钮 + 右导航「存档」是 **空壳** | 需接面板 |

**结论**：zhushen-space 已经有「单存档自动持久化」（所有 `drpg-*`），缺的是**命名多槽**这一层，且**对话历史没落库**，必须先补上。

## 3. 设计方案

### 3.1 一个存档槽包含什么
```ts
interface SaveSlot {
  id: string;            // "slot_<时间戳>"
  name: string;          // 存档名（可改）
  createdAt: number;
  updatedAt: number;
  // 预览信息（列表展示用）
  preview: {
    turn: number;        // 回合数
    playerName: string;  // 主角名
    location: string;    // 当前世界/位置
    lastText: string;    // 最近一段正文摘要
  };
  // 完整数据
  data: {
    stores: Record<string, unknown>;  // 所有 drpg-* 的快照（key→JSON）
    messages: ChatMessage[];          // 对话历史
  };
}
```

### 3.2 存哪
- **推荐 IndexedDB**（对齐源前端，容量大）：用一个极简原生封装或 `idb-keyval`（轻依赖）。
- 折中：**元信息存 localStorage、完整 data 存 IndexedDB**（列表加载快）。
- 最简（如果存档不大）：全放 localStorage（`drpg-saves`）。**不推荐**——多存档 + 全对话会超 5MB。

本指导用 **IndexedDB（原生封装，零依赖）**。

### 3.3 关键难点：对话历史先持久化
`messages` 现在是 `App` 的 `useState`，存档读不到也存不住。两种解法（二选一）：
- **A（推荐）**：把 `messages` 抽成 `chatStore`（Zustand + persist 到 `drpg-chat`）。好处：① 自动并入 `drpg-*` 快照；② **顺便修复刷新丢失对话**；③ 叙事记忆/总结也能跨刷新存活。
- B：保持 useState，由 `App` 把 `messages` + `setMessages` 传给存档面板，存/读时手动桥接。

下面按 **A** 写。

## 4. 实现步骤

### 步骤 ① 对话抽成 `chatStore`（`src/store/chatStore.ts`）

```ts
import { create } from 'zustand';
import { persist } from 'zustand/middleware';

export interface ChatMessage {
  id: number;
  role: 'user' | 'assistant';
  content: string;
  smallSummary?: string;
  largeSummary?: string;
}

interface ChatState {
  messages: ChatMessage[];
  nextId: number;
  setMessages: (m: ChatMessage[] | ((prev: ChatMessage[]) => ChatMessage[])) => void;
  pushMessage: (m: Omit<ChatMessage, 'id'>) => number;
  clear: () => void;
}

export const useChat = create<ChatState>()(
  persist(
    (set, get) => ({
      messages: [],
      nextId: 1,
      setMessages: (m) => set((s) => ({ messages: typeof m === 'function' ? (m as any)(s.messages) : m })),
      pushMessage: (msg) => {
        const id = get().nextId;
        set((s) => ({ messages: [...s.messages, { ...msg, id }], nextId: id + 1 }));
        return id;
      },
      clear: () => set({ messages: [], nextId: 1 }),
    }),
    { name: 'drpg-chat' },
  ),
);
```

> `App.tsx` 里把本地 `const [messages, setMessages] = useState<ChatMessage[]>([])` 换成 `const messages = useChat(s=>s.messages); const setMessages = useChat(s=>s.setMessages);`，`ChatMessage` 接口改从 `chatStore` 导入。其余 `setMessages((prev)=>…)` 用法保持兼容。

### 步骤 ② IndexedDB 极简封装（`src/systems/saveDb.ts`）

```ts
const DB = 'drpg-archive';
const STORE = 'slots';

function openDb(): Promise<IDBDatabase> {
  return new Promise((res, rej) => {
    const req = indexedDB.open(DB, 1);
    req.onupgradeneeded = () => {
      if (!req.result.objectStoreNames.contains(STORE)) req.result.createObjectStore(STORE, { keyPath: 'id' });
    };
    req.onsuccess = () => res(req.result);
    req.onerror = () => rej(req.error);
  });
}
async function tx<T>(mode: IDBTransactionMode, fn: (s: IDBObjectStore) => IDBRequest): Promise<T> {
  const db = await openDb();
  return new Promise((res, rej) => {
    const r = fn(db.transaction(STORE, mode).objectStore(STORE));
    r.onsuccess = () => res(r.result as T);
    r.onerror = () => rej(r.error);
  });
}
export const saveDb = {
  put: (slot: any) => tx('readwrite', (s) => s.put(slot)),
  get: (id: string) => tx('readonly', (s) => s.get(id)),
  all: () => tx<any[]>('readonly', (s) => s.getAll()),
  del: (id: string) => tx('readwrite', (s) => s.delete(id)),
};
```

### 步骤 ③ 存档管理逻辑（`src/systems/saveManager.ts`）

```ts
import { saveDb } from './saveDb';
import { useChat } from '../store/chatStore';

// 需要纳入快照的所有 store key
const STORE_KEYS = [
  'drpg-save', 'drpg-settings', 'drpg-items', 'drpg-player-evo',
  'drpg-npc', 'drpg-npc-evo', 'drpg-characters', 'drpg-memory', 'drpg-misc', 'drpg-chat',
];

export interface SlotMeta { id: string; name: string; createdAt: number; updatedAt: number; preview: any }

function snapshotStores(): Record<string, unknown> {
  const out: Record<string, unknown> = {};
  for (const k of STORE_KEYS) {
    const v = localStorage.getItem(k);
    if (v != null) out[k] = v;          // 直接存原始 JSON 字符串，回写最稳
  }
  return out;
}

export async function createOrUpdateSlot(id: string | null, name: string) {
  const msgs = useChat.getState().messages;
  const last = [...msgs].reverse().find((m) => m.role === 'assistant')?.content ?? '';
  // 预览信息（按需从各 store 取）
  const preview = {
    turn: msgs.length,
    playerName: readJson('drpg-player-evo')?.state?.profile?.name ?? '主角',
    location: readJson('drpg-misc')?.state?.worldName ?? '',
    lastText: last.replace(/<[^>]+>/g, '').slice(0, 80),
  };
  const now = Date.now();
  const realId = id ?? `slot_${now}`;
  const existing = id ? await saveDb.get(id) : null;
  await saveDb.put({
    id: realId, name, createdAt: existing?.createdAt ?? now, updatedAt: now, preview,
    data: { stores: snapshotStores() },   // messages 已在 drpg-chat 里
  });
  return realId;
}

export async function listSlots(): Promise<SlotMeta[]> {
  const all = await saveDb.all();
  return all.map((s) => ({ id: s.id, name: s.name, createdAt: s.createdAt, updatedAt: s.updatedAt, preview: s.preview }))
            .sort((a, b) => b.updatedAt - a.updatedAt);
}

export async function loadSlot(id: string) {
  const slot = await saveDb.get(id);
  if (!slot) return false;
  for (const [k, v] of Object.entries(slot.data.stores)) {
    if (typeof v === 'string') localStorage.setItem(k, v);
  }
  // 最稳：写回 localStorage 后整页重载，让所有 zustand persist 重新 hydrate
  location.reload();
  return true;
}

export async function deleteSlot(id: string) { await saveDb.del(id); }

export async function exportSlot(id: string) {
  const slot = await saveDb.get(id);
  if (!slot) return;
  const blob = new Blob([JSON.stringify(slot, null, 2)], { type: 'application/json' });
  const url = URL.createObjectURL(blob);
  const a = document.createElement('a');
  a.href = url; a.download = `${slot.name || 'save'}-${slot.id}.json`; a.click();
  URL.revokeObjectURL(url);
}

export async function importSlot(raw: string) {
  const slot = JSON.parse(raw);
  if (!slot?.data?.stores) throw new Error('存档格式不正确');
  slot.id = `slot_${Date.now()}`;          // 导入生成新 id，避免覆盖
  slot.updatedAt = Date.now();
  await saveDb.put(slot);
  return slot.id;
}

function readJson(k: string): any { try { return JSON.parse(localStorage.getItem(k) ?? ''); } catch { return null; } }
```

> **回写策略用「整页 reload」**：Zustand `persist` 在页面加载时 hydrate。读档时把各 `drpg-*` 写回 localStorage 再 `location.reload()`，是最省心、最不易出 bug 的方案（避免逐 store 手动 `setState` 的遗漏与中途状态不一致）。

## 5.5 实际实现（已落地）

落地时做了一处关键调整：**没有把对话抽成 chatStore**（流式逐 token 写 localStorage 会卡），改用 **reload + sessionStorage** 方案，更稳：

- `systems/saveDb.ts`：原生 IndexedDB 封装（`drpg-archive` 库，`slots` 表，keyPath=id）。
- `systems/saveManager.ts`：
  - `saveSlot(id|null, name, messages)`：快照全部 9 个 `drpg-*`（直接搬 localStorage 原始 JSON）+ `messages`（由面板传入）+ 预览（回合/主角名/世界/最近正文）→ 存 IndexedDB。
  - `loadSlot(id)`：把快照写回 localStorage → 对话历史暂存 `sessionStorage(drpg-pending-chat)` → `location.reload()`。**reload 让 zustand-persist 各 store 与 gameStore（模块初始化时读 localStorage）一并恢复**——因为 `gameStore` 用的是手写持久化、没有 `persist.rehydrate()`，逐 store rehydrate 不可靠，整页 reload 最稳。
  - `listSlots / renameSlot / deleteSlot / exportSlot（下载.json）/ importSlot（白名单校验+生成新 id）`。
- `App.tsx`：挂载时 `useEffect` 从 `sessionStorage` 取回 `drpg-pending-chat` → `setMessages` + 同步 `msgId`，并据 `drpg-pending-started` 进入游戏。`messages` 仍是 `useState`（保存时由面板快照，不持久化、不影响流式性能）。
- `components/SaveLoadPanel.tsx`：存档列表（名称/时间/回合/主角/位置/最近正文预览）+ 新建/覆盖/读取(二次确认)/改名/删除/导出/导入。
- 入口：右导航 **💾 存档**；开始界面 **读取存档** 热区（读档后经 reload 自动进入游戏）。

### 对话跨刷新自动保留（已补，贴近原版增量写法）

后续补做了对话自动持久化（参考 fanren-remake 的 `archiveChatMessages` 增量写）：

- `systems/chatDb.ts`：对话存独立 IndexedDB 库 `drpg-chat`，**每条消息一行**（keyPath=id）。`putChanged` 用内存 `lastJson` 缓存做 **diff，只写内容变化/新增的消息、删除已移除的**——流式时只有"正在生成那条"在变 → **每 token 只写 1 行，不卡**。`loadAll`/`replaceAll`/`clearAll`。
- `App.tsx`：挂载 `useEffect` → `chatDb.loadAll()` 回填 `messages` + 同步 `msgId`（`chatHydrated` 标志防止 hydrate 前误写覆盖）；`messages` 变化时 `chatDb.putChanged(messages)` 增量落库。
- `saveManager.loadSlot`：改为 `replaceChat(slot.data.messages)` 写回 chatDb（不再用 sessionStorage 搬运），reload 后 `App` 从 chatDb 读回。
- 效果：**刷新/重开网页，对话自动还在**（回到 StartScreen 点「开始游戏」即接着上次）。`messages` 仍是 `useState`，只是多了一层 IndexedDB 镜像。

## 6. 注意事项

- **必须先持久化对话**（步骤①），否则存档读不到正文历史。
- **快照用原始 JSON 字符串**（直接搬 localStorage 的值），读档回写后 `reload` 让 persist 自然 hydrate——别尝试逐字段反序列化再 setState，易漏字段、易状态不一致。
- **版本兼容**：存档里 `data.stores` 是各 store 当时的 persist 结构；若以后 store 结构大改，各 store 的 `merge` 回落逻辑（你现有的）能吸收大部分差异。可在 slot 里加 `appVersion` 备查。
- **导入安全**：导入的 JSON 直接写库，建议校验 `data.stores` 存在且 key 在白名单内，避免脏数据。
- **容量**：IndexedDB 足够；若坚持 localStorage，多存档 + 全对话很快超 5MB 报错。
- **「自动存档」**：现有 `drpg-*` 持久化本身就是一份"当前进度"。多槽是在它之上做命名快照；读档=用某快照覆盖当前 `drpg-*` 再 reload。
- 与「叙事记忆/生平压缩/杂项」无冲突：它们的数据都在 `drpg-memory`/`drpg-misc`/`drpg-chat` 里，会一并进存档。
