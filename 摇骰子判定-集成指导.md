## 0. 核心理念

一句话：**玩家手动发起检定 → 前端纯计算掷骰 + 定成功等级 → 结果在发送前注入主提示词 → AI 本回合服从骰子续写。**

与酒馆原版的差异：zhushen 是"一次调用生成整回合正文"，没有"AI 边写边停下等掷骰"的能力。所以采用**手动触发 + 发送前注入**，让骰子在**当回合**就影响剧情。

---

## 1. 整体交互模型（手动触发）

两条输入路径并存：

```
不想 roll → 照常在主输入框打字 → 直接发送（全程不碰骰子）

想 roll  → 点右侧导航 🎲 打开骰子页 DicePanel
           ├─ 填本回合行动（如"我撬开这扇上锁的铁门"）
           ├─ 选检定类型（快捷下拉：开箱/说服/求爱/战斗… 自动带出属性+是否对抗+好感项）
           ├─ 选属性(六维下拉) + 选难度(简单~极难)；对抗则选对手(在场NPC或强度档)
           │   └─ 或点「✨AI建议」：可选的小调用回填 属性/难度/相关技能天赋（可改）
           ├─ (可选)勾相关技能/天赋（默认按名称自动勾）
           ├─ 【掷骰】→ 摇骰动画 → 结果卡：总值/DC/成功率P%/成功等级/配色
           └─ 【注入并发送】→ 把"行动文本 + <检定结果>块"作为本回合输入发出
```

**注入给主 AI 的内容**（发送前，本回合生效）：

```
我撬开这扇上锁的铁门。
<检定结果> B1 撬锁(敏捷) 难度=困难 → 普通成功(d20:8 + MOD:10 = 18 ≥ DC16) </检定结果>
```

AI 看到结果，本回合正文就服从骰子。结果同时存 `ChatMessage.diceChecks[]` 渲染成内联骰子卡。

**关键原则：语义相关性交给（可选的）AI 判断，具体加几分由前端查表确定。** AI/玩家只选"哪个属性/技能/天赋相关"，前端去 `characterStore` 查层阶/评级套**固定数值表**算分 —— 数值 100% 可复现，AI 不能瞎编分数。

## 2. 数值引擎（相对修正，适配开放六维）

**为什么不能照搬 CoC/D&D 原值**：zhushen 六维是**开放数值**（默认 5，`trueAttr` 每 80 普通 = 1 真实，后期上千）。CoC 的 `1d100≤属性`、D&D 的 `属性调整=(属性-10)/2` 都假设属性在固定小区间，直接搬会崩（前期必败、后期必成、骰子变噪声）。

**解法**：所有修正都**相对化**（相关属性 vs 角色自身六维均值），绝对数值不进阈值。难度由 AI/玩家**相对该角色**设定。这样一阶撬一阶锁、九阶撬九阶锁都能是"普通"。

### 普通检定（百分骰口径，d20 见 §3）

```
P = clamp( 基础难度率 + 属性修正 + 技能修正 + 天赋修正 + 状态修正 + 幸运修正 , 5 , 95 )
成功 = (1d100 掷点 ≤ P)
```

| 修正项 | 规则 |
|---|---|
| 基础难度率 | 见 §6 难度表 |
| 属性修正 | `clamp(round((相关属性 − avg)/avg × 30), −15, +20)`，avg = 六维均值 |
| 技能修正 | **全部技能**各按 `Skill.level` 层阶求和、封顶 d20±10/d100±40：入门+5/精通+10/大师+15/宗师+20/极道+25 |
| 天赋修正 | **全部天赋**正负相加、封顶 d20±12/d100±48，按 `Trait.rarity`：D+3/C+6/B+9/A+12/S+16/SS+21/SSS+27/负面−12 |
| 装备修正 | 已装备物品按属性相关性×品质(grade)：武器→力/敏、防具→体质、饰品/特殊→智/魅/幸，封顶 d20±6/d100±30 |
| 状态修正 | 扫 `statusEffects`+列4，合计 clamp(±30)：buff+6(强+10)/debuff−8(重−12)/中性0 |
| 幸运修正 | `clamp(round((luck − avg)/avg × 12), −8, +12)` |

**成功等级**：roll≤5 大成功 / ≤P/5 极难成功 / ≤P/2 困难成功 / ≤P 普通成功 / >P 失败 / ≥96 大失败。

## 3. 骰子双模式（DND d20 默认 / CoC 百分骰）

**同一套相对修正引擎**，只换"骰子 + 目标值 + 暴击"层。在 ROLL 设置页切换，**默认 d20**。

### 3a. DND d20 模式（默认）

```
总值 = 1d20 + MOD      总值 ≥ DC → 成功
```

**DC 由难度决定**（可调）：简单10 / 普通13 / 困难16 / 极难20 / 几乎不可能25。

**MOD = 各修正之和**（同样相对逻辑，压到 d20 尺度）：

| 修正项 | d20 取值 |
|---|---|
| 属性 | `clamp(round((相关属性 − avg)/avg × 6), −4, +5)` |
| 技能 | 入门+1 / 精通+2 / 大师+3 / 宗师+4 / 极道+5 |
| 天赋 | D·C+1 / B+2 / A+3 / S+4 / SS+5 / SSS+6 / 负面−2 |
| 状态 | buff+1(强+2) / debuff−1(重−2)，合计 ±4 |
| 幸运 | `clamp(round((luck − avg)/avg × 2), −1, +2)` |
| 好感(社交) | 敌对−4 / 冷淡−2 / 中立0 / 友好+2 / 亲密+4 / 挚爱+6 |

**招牌机制**：
- **自然 20**（骰子原始点=20）→ 大成功，无视 DC + 后果翻倍
- **自然 1** → 大失败，无视 MOD，自动失败 + 可能反噬
- 总值超 DC ≥10 → 碾压；勉强够 → 险过
- **优势 = 2d20 取高，劣势 = 2d20 取低**（取代 CoC 奖惩骰，顺势/逆势时用）
- d20 模式仍反算并显示 P% 胜算：`P = clamp((21 − (DC − MOD))/20 × 100, 5, 95)`

### 3b. CoC 百分骰模式（备选）

即 §2 的 `1d100 ≤ P`，成功率直读为百分比，成功等级按 margin 分级。保留它几乎零成本，给想要纯数值化判定的场景。

## 4. 对抗 / 对战（必须算敌方属性）

普通检定是"人 vs 固定难度"；对战是"人 vs 会抵抗的对手"（战斗命中/闪避、掰手腕、心理博弈）。

**对战 = 绝对强度差 + 相对修正差**（单骰，1 次掷骰）：

```
百分骰: P = clamp(50 + 强度差 + (我方相对修正 − 敌方相对修正), 5, 95)；1d100 ≤ P → 胜
d20:   总值 = 1d20 + 我方MOD + 强度差d20 ；DC = 13 + 敌方MOD ；总值 ≥ DC → 胜
```

> 为什么单列"强度差"：纯相对修正（属性 vs 自身均值）在"人 vs 任务"成立，但"人 vs 人"会丢掉绝对强弱 —— 全 2 的杂鱼和全 500 的神，属性修正都≈0。所以对战必须 = 绝对强度差（等级/阶位/强度档差）+ 相对修正差（谁更专精）。

- **相对修正** = 各自的 (属性+技能+天赋+状态+幸运)，按 §2/§3 表，各自相对自身均值。
- **强度差** = `clamp((我方强度分 − 敌方强度分) × 0.6, −40, +40)`；d20 版 `clamp(round((我方强度分 − 敌方强度分)/8), −6, +6)`。强度分查下表。

**敌方强度档表**（未建档敌人在骰子页直接选；建档 NPC 由其 Lv/阶位/bioStrength 自动取）：

| bioStrength | 阶位参考 | 强度分 |
|---|---|---|
| T0 杂鱼 | 凡人/炮灰 | 0 |
| T1 | 一阶 | 10 |
| T2 | 二阶 | 20 |
| T3 勇士 | 三阶 | 30 |
| T4 | 四阶 | 40 |
| T5 | 五阶 | 50 |
| T6 | 六阶 | 60 |
| T7 | 七阶 | 70 |
| T8 | 八阶 | 80 |
| T9 源初 | 九阶+ | 90 |

每差 1 档 ≈ 百分骰 ±6 / d20 ±1；档差越大越主导，封顶 ±40（百分骰）/ ±6（d20），给弱方留一线（靠 clamp 下限 5% / 自然20 翻盘）。

- **我方/敌方可选不同属性**：力量挥砍 vs 敏捷闪避；魅力游说 vs 智力识破。
- **敌方属性来源**：对手选择器**只列在场 NPC**（`onScene && !isDead`，皆建档），读其 `npcStore.attrs` 六维 + `characterStore[id]` 技能天赋 + `npc.items` 装备 + `favor` 好感，算敌方相对修正；强度分由其 Lv/阶位/bioStrength。（引擎仍保留 `STRENGTH_TIERS`，未来需要未建档敌人时再启用。）
- **多打一 / AOE**：取**强度分最高**的敌人当代表，**只掷一次**；胜则视为压制其余，由正文 AI 分配结果。

### 4a. 未登场 / 下回合才出现的敌人

本质 = "未建档敌人"特例（强度档兜底）+ 时序问题。zhushen 里 NPC 在正文之后的**登场判断/六维生成 post-phase** 才建档，所以敌人可能"出现在正文里但还没 Cx 档案"。分三种：

- **① 本回合刚冒头、还没档案**：骰子页用「**临时敌人**」模式 —— 填名（或"未知黑影"）+ 选**强度档**（或点「✨AI建议」让它从正文铺垫估档），按强度分掷。结果注入本回合 → AI 叙述敌人现身 + 结果 → post-phase 登场判断再建真实档案。
- **② 敌人下回合才真正登场（你这回合只是预备/铺垫）**：**别急着掷战斗骰**。这回合只掷"你正在做的动作"（察觉/预判/布陷 = 普通检定）；敌人由 AI 叙述登场，等它真在场了（下回合，已建档、`genVariedAttrs`/`autoGenMissingAttrs` 已按阶位/职业/bioStrength 自动生成六维），再对它掷对抗。
- **③ 敌人下回合突袭你（敌方主动）**：你无法预掷；等突袭发生那回合再掷（防御/闪避对抗），敌人此时已登场建档或用强度档兜底。

**一致性兜底（推荐）**：骰子页对"临时敌人"选了强度档时，前端**预埋一个 NPC stub**（`onScene=false` 占位 + 记下 tier）传给登场判断当 hint，让真实档案的 bioStrength/六维按这个档生成 —— 避免"我按 T5 Boss 掷的、结果档案生成成 T2"的不一致。

**原则**：别对"你还不确定存不存在"的敌人掷战斗骰；掷你这回合实际做的动作，敌人登场后再做战斗判定。强度档就是用来填"敌人在场但还没建档"这段空窗的。

> 备选 Approach 2：双骰各算 P/总值比成功等级，平局比余量。更像正统 CoC/D&D 对抗，但要掷两次。先用单骰。

## 5. 杂项检定映射表（不是新系统，是映射）

开箱/说服/求爱全都套**普通检定或对抗检定**，只需查表决定四件事：**用哪个属性 / 是不是对抗 / 有没有好感修正 / 后果挂哪**。

| 检定类型 | 方式 | 我方属性 | 对方抵抗属性 | 特殊修正 | 常见相关技能/天赋 | 典型后果 |
|---|---|---|---|---|---|---|
| 开箱/撬锁 | 普通 vs 难度 | 敏捷(手法)/智力(机关) | — | — | 盗贼技巧·机关术·巧手 | 大成功额外掉落；失败触发陷阱；大失败锁死 |
| 解除陷阱 | 普通 | 敏捷/智力 | — | — | 拆解 | 大失败陷阱触发 |
| 潜行/躲藏 | 对抗 | 敏捷 | 智力(感知) | 暗处/掩体+ | 隐匿 | 失败被发现→进战斗 |
| 察觉/搜查 | 普通 | 智力 | — | — | 侦查 | 成功得线索 |
| 说服/谈判 | 对抗 | 魅力 | 智力(识破) | 好感± | 话术·外交 | 立场改变，favor± |
| 威吓 | 对抗 | 力量或魅力 | 体质/魅力 | 实力差·好感− | 威压 | 屈服或激怒 |
| 欺骗/说谎 | 对抗 | 魅力或智力 | 智力(看穿) | — | 欺瞒 | 拆穿则 favor-- |
| 求爱/勾引 | 对抗+好感门槛 | 魅力 | 智力(理智) | **好感主导** | 魅惑·风情·共情 | 关系推进，favor 大幅± |
| 砍价/交易 | 对抗 | 魅力 | 魅力/智力 | 好感± | 商术 | 价格浮动 |

**六维无"感知/意志/理智"独立项** → 统一用 **智力(int)** 兼任心智/识破/理智、**魅力(cha)** 兼任意志/定力。

**社交/求爱的好感度修正**（接 `npcStore` favor）：敌对−20 / 冷淡−10 / 中立0 / 友好+10 / 亲密+20 / 挚爱+30（d20 模式 ±4/±2/0/+2/+4/+6）。**求爱好感过低（<20）直接极难或必败**，防一上来硬求爱靠运气成功。

## 6. 难度设定（可选 + 可调）

四档（可选第五档），骰子页顶部下拉选，默认**普通**，也可被「✨AI建议」回填。基础率/DC 在 ROLL 设置页可编辑。

| 难度 | 百分骰基础率 | d20 DC |
|---|---|---|
| 简单 | 85 | 10 |
| 普通（默认） | 65 | 13 |
| 困难 | 45 | 16 |
| 极难 | 25 | 20 |
| 几乎不可能（可选） | 10 | 25 |

> 难度是**"对该角色而言"的相对难度**，不是绝对强度。对抗检定时这栏换成敌方优势值/MOD。

## 7. 后果系统（挂现有短指令 + 多面骰数值）

检定定成败，**d4~d12 多面骰掷后果随机数值**，全部接现有短指令，不另造：

| 场景 | 骰子 → 指令 |
|---|---|
| 大成功开箱 | 掉落 `2d6 × 100` 乐园币 → `transferSpiritStones` |
| 触发陷阱 | 受伤 `3d6` → `hp.B1 -= N` / `addStatus(...)` |
| 求爱成功 | 好感 `+1d10` → `character.<id>.stats.favor += N` |
| 物品获得 | `createItem(...)` |

**暴击后果倍率**（成功等级缩放后果骰/数值）：

| 成功等级 | 倍率 | 额外 |
|---|---|---|
| 大成功（自然20 / CoC≤5） | ×2 | 战斗=要害/斩杀判定；社交=好感跃迁+额外承诺；开箱=多一件稀有掉落；潜行=完全无声+情报；常附免消耗/增益 |
| 碾压成功（超 DC ≥10 / CoC≤P/5） | ×1.5 | 干净利落、附带小优势 |
| 成功 | ×1.0 | 正常 |
| 失败 | ×0 | 行动未成，至多轻微代价 |
| 大失败（自然1 / CoC≥96） | 反噬（约 ×1，指向自己） | 战斗=破绽被反击/武器损坏；社交=好感暴跌/结仇；开箱=陷阱触发+锁死；潜行=直接暴露+敌方先手 |

> 倍率乘在后果骰/数值上（战斗基础伤 `3d6`，大成功 = `3d6 × 2`；求爱好感 `+1d10`，大成功 = `+1d10 × 2`）。前端可直接掷好后果骰 × 倍率、生成对应短指令，并把幅度写进 `<检定结果>` 让 AI 照此叙述。大失败把后果翻向自己（反噬）。

骰子页另给**自由骰输入框**：直接打 `2d6+3`、`4d8`、`1d20`、`1d100` 掷任意多面骰（GM 临时裁定 / 玩家自掷）。关系推进、立场转变等剧情后果交给正文 AI（它本回合看到 `<检定结果>` 会自然写）。

## 8. 摇骰动画（纯 CSS/SVG，不加依赖）

- **d20**：SVG 二十面体轮廓（六边形外框 + 内部三角面），点 ROLL 后 die 摇晃/旋转（CSS keyframes：`rotate + scale + translate`），中心数字快速滚动随机 1–20（~700ms，`requestAnimationFrame` 或 setInterval），缓动减速后停在结果。
- **暴击反馈**：自然 20 → 金色描边 + 光晕脉冲 + "大成功！"；自然 1 → 红色裂纹闪烁 + "大失败！"。
- **百分骰**：数字翻滚动画（快速随机 → 减速停在掷点），或滚动的进度刻度。
- **多面骰**：通用"数字滚动"动画即可，骰面形状按 d4/d6/d8/d10/d12 切 SVG 轮廓。
- **优势/劣势**：并排掷两颗 d20，高/低那颗高亮、另一颗灰掉。
- **动画常驻，不做"跳过动画"开关**（按用户要求）；时长可在设置里调（默认 ~760ms）。
- 实现位置：`DiceRoller.tsx`（动画组件，被 `DicePanel` 和 `DiceCard` 复用）。

## 9. 独立 API + ROLL 点设置页

**先澄清**：掷骰是**纯前端确定性计算，不花任何 API**。API 只服务骰子页那个**可选的「✨AI建议属性/难度」按钮**。

完全套现有独立 API 架构（同物品/主角/NPC/杂项…）：

- **数据**：`diceStore` 加 `diceApi` + `diceUseSharedApi`（仿 `miscStore`/`territoryStore`）。
- **路由**：新增 `featureKey: 'dice'` 进 `settingsStore.apiRoutes`，走 `resolveApiChain('dice', diceApi)` + `apiChatFallback`（多接口轮流 + fallback，复用中心 API 接口库）。
- **入口**：**设置 → 变量管理 → 🎲 ROLL点设置**（仿 `MiscManager`），两 Tab：

| Tab | 内容 |
|---|---|
| 检定设置 | 总开关 / 骰子模式(d20·CoC) / 四档难度基础率·DC(可编辑) / 各修正项上下限 / 求爱好感门槛 / 动画开关与时长 / 是否需手动点"投骰" |
| API 设置 | 「共用主 API」或「独立配置」 + `ApiRoutePicker`（多接口选择+排序）+ 测试连接 / 拉取模型 |

## 11. MVP 分阶段

1. **MVP**：`diceStore` + `diceEngine`（d20 模式 + 相对修正）+ `DiceRoller` 动画 + `DicePanel`（普通检定，手动选属性/难度）+ 结果注入 + `DiceCard` 渲染。跑通 掷骰→注入→AI服从 闭环。
2. **V2**：对抗检定（敌方属性 + 多打一取最强）+ 杂项快捷下拉 + 好感修正 + 「✨AI建议」调用（独立 API）。
3. **V3**：CoC 百分骰模式切换 + 多面骰后果/自由骰 + 优势劣势 + 暴击动画细化 + 设置页全量可调 + 检定历史。

## 12. 关键风险

1. **因果方向**：必须"发送前注入"或"下回合结算"，事后掷骰 = 白掷。本设计用发送前注入。
2. **数值缩放**：别用绝对值，相对修正 + clamp[5,95]/d20，保证骰子永远有意义。
3. **AI 不守格式**：靠鲁棒正则 + 代码注入铁则 `DICE_RULE` 兜底（借鉴源系统 `normalizeCheckSuggestionCommandInput`）。
4. **API 成本**：仅「AI建议」按钮用，可关、低频、走独立廉价模型。
5. **属性清单要喂给「AI建议」调用**，否则它瞎编属性名（用现成 `buildStructuredRecall`）。
