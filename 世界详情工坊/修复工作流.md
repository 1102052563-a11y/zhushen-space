# 世界详情工坊 · 表驱动修复工作流

> 用途：对已有产出里"字数不够 / 灌水套话 / 跨世界雷同 / 结构异常"的文件做**逐个修复**。
> 与"生成新世界"是两码事——本流程只**改已存在的 md**，不新建世界、不碰 manifest/批次表/脚本。
> 金标准写法仍看 `产出/批次02/诡秘之主.md`、`样板/我欲封天.md`；内容铁则仍看 `README.md`。

---

## 0. 唯一真相：修复清单表

- **机器可读**：`清单/修复清单.json`（每文件一行，含 `priority/batch/file/name/lib/issue/status`）
- **人类可读**：`修复清单.md`（按 P0/P1/P2 分组的表格，状态列 ⬜待修复 / ✅已修复）

每次要修，**先看这张表**，从 `status=待修复` 里按优先级取文件。**不要**自己另找文件、不要凭记忆改。

**优先级**：
| 级 | 含义 | 处理 |
|---|---|---|
| **P0** | 机检不过关：剧情/切入点字数不足、缺必备段落、结构异常(两世界拼接/重复## 段)、世界名不在 manifest | **必须修**，不修则该条目在游戏里不可用 |
| **P1** | 灌水标记(【扩写·】【加厚·】【原作信息增密·N】…) / 跨世界雷同大段(多个世界共用同一段套话) | 内容质量，需替换成该世界**独有真实内容** |
| **P2** | 文件名与标题不符(多为全角/半角符号) / 软警告(来源<3、被封印式措辞) | 可选，顺手修 |

## 1. 铁则（修复时同样适用，违反=返工）

1. **两种角色，权限不同**：
   - **编排者（顶层）**：可以一次派出**最多 5 个** agent 并行修复，每个 agent 拿一份**互不重叠**的文件清单（用 `scripts/_split_batches.mjs` 自动切分，见 §4）。派完等它们全部回报，核实 ✅ 数，**立即开始下一轮，不需要向人确认**。
   - **修复 agent（被派出的）**：**自己动手**，一次只修一个文件，修完打标志、机检过，再取自己清单里的下一个。**修复 agent 严禁再派生任何子agent**——历史上层层转包的执行者产出全部为空。必须用 Read/WebFetch/WebSearch/Edit/Write/PowerShell 亲手改。
2. **人物真名**，搜不到写"不详"，不许用"群像/原作主角/核心道侣/牙人"等占位符当人名。
3. **禁止再灌水**：绝不重新引入 `【扩写·N】【加厚·N】【原作信息增密·N】【原作细节增补·N】【独有细描·XX·N】【细目N】` 这类编号复读段；绝不逐句复制粘贴；绝不写"跨媒介流行作品""可被契约者切入的完整任务世界"这类跨世界通用套话。字数不够=**再检索真实剧情/人物/宝物/名场面**补，不是灌水。
4. **字数是硬门槛，不达标不许收工**：主库剧情**必须 ≥10000 字**、切入点 ≥1500 字；休闲剧情必须 ≥6000 字。差多少就继续补多少，**不存在"实在补不动就算了"这个选项**——机检不过就是没修完，必须回去继续检索、继续补**有意义的真实内容**，直到 `_verify_and_mark.mjs` 给出 ✅。
   **但补的内容必须有意义**（下面这些都是合法的加字方向，按顺序挖，总能挖够）：
   - 逐卷／逐章／逐关卡展开主线剧情的具体经过与因果
   - 每个具名人物的身份、性格、能力装备、人物弧光、与他人的立场关系（配角也写全）
   - 力量／等级体系**逐级**列全，每级配战力表现描述
   - 地理舞台逐地点说明（每处的功能、发生过什么事）
   - 各势力的宗旨、地盘、具名代表人物、彼此恩怨
   - 关键宝物／道具的真名、来历、能力、剧情下落
   - 名场面的前因后果、伏笔与回收
   - 编年式大事记时间线
   - 不同改编版本（原作／动画／漫画／游戏／剧场版）的剧情差异
   - 结局走向与后日谈
   **绝不允许**用重复段落、通用套话、元写作说明（"本段用于…""写作时应…"）、编号复读段来凑字数——那是灌水，机检和人工复核都会打回。
5. **补字数的唯一合法来源**＝该世界**真实**的分卷剧情、具名人物弧光、势力互动、宝物来历、名场面因果、结局方向。联网检索优先：
   - 中文网络小说：`https://www.sobqg.com/searchBook.html?keyword=<书名URL编码>` → 进书页 → 免费章节核对。
   - 影视/游戏/动漫/神话：百度百科 / 维基百科 / 萌娘百科 / Fandom wiki / Steam / VNDB / 官网。
   - 至少实际打开 ≥3 个真实 https 页（`## 来源` 里挂上）。查不到写"不详"，**禁止编造**。
6. **阶位覆盖必须连续、必须从一阶起（有高阶就必须有全部低阶）**：
   一个世界既然存在高阶切入点，就必然也存在所有低阶切入点——**绝不允许只写高阶**（如只有八阶、只有四阶、或"三阶+九阶"跳着写）。
   `## 阶位切入点` 段必须把 **一阶 ~ 该世界最高阶** 的每一阶都写全，每阶七字段齐全、各阶内容互不重复（禁止各阶复制同一句）。
   该世界的最高阶查 `清单/manifest.json` 的 `tiers`（已统一补齐为连续区间）。机检会把缺失阶位判为**错误**并拦截，缺哪阶就补哪阶。
   低阶怎么写：低阶不是"弱化版高阶"，而是**该世界在主角尚弱时真实存在的处境**——市井/底层视角、还没卷入主线时的边缘事件、被高阶存在的余波波及、给强者跑腿打杂、在原作主角成名前的时间点切入等，都要用该世界真实的地名、势力、具名人物来写。
7. **顶点铁则**：超阶巅峰＝存在·情报优先/条件性胜利。通篇（含提示句）**不要出现"被封印"三个字**——机检是字面子串匹配，连"严禁写成被封印式解释"都会误报；改用"严禁简化为单纯的'战力归零'式解释"。
8. **跨世界重复(P1)修法**：这类是多个世界共用同一段套话（如假面骑士各作共用"在XXX的「阶段」…可介入事件示例：①护送一批硬通货…"、或休闲批次共用"第一周不要碰评估印…"）。要给**当前这个世界**写它**专属**的阶段/事件/角色细节，把那段共用套话整段换掉，不能只改开头保留主体。
9. **休闲轨道**（lib=休闲）走 `模板/休闲剧情模板.md`+`休闲切入点模板.md`，剧情门槛≥6000字，**无战力/阶位措辞**；主库走战斗模板，剧情≥10000、切入点≥1500、需"乐园阶位映射"+"阶位↔"行。查 `清单/manifest.json` 确认 lib。

## 2. 单文件修复流程（对表里每个 待修复 项重复）

```
① 从 修复清单.md 取一个 status=待修复 的文件（P0 优先，再 P1，再 P2）
② Read 打开该文件，看现状
③ node scripts/compile-worldbook.mjs --check "产出/批次NN/<世界名>.md"
   → 看具体差多少字、缺哪段、报什么错
④ 联网检索该世界真实资料（见铁则4），Edit/Write 扩写/替换：
   · 字数不够 → 补真实剧情线/人物/宝物段落到超标 10~15%（战斗≥11000、休闲≥6800）
   · 有灌水标记 → 删掉那些编号复读段，换成真实内容
   · 跨世界重复 → 把共用套话整段改写为本世界独有细节
   · 结构异常(两世界拼接/重复##段) → 删掉不属于本世界的段落、合并重复的##段
   · 文件名不符 → 一般是标题里全角/半角符号问题；以 manifest 里的规范名为准改 # 标题
⑤ node scripts/_verify_and_mark.mjs "产出/批次NN/<世界名>.md"
   → ✅ 通过则自动盖「已修复」标志(<!--repaired-->)；❌ 则按提示继续补，别硬凑
⑥ 过关后回到 ①，取下一个
```

## 3. 「已修复」标志与状态同步

- **标志**：文件 meta 行下方的 `<!--repaired: 20260721-->`。由 `_verify_and_mark.mjs` 在**通过全部门禁后**自动插入——不是手写、不是agent口头声称，是脚本验证过的。
- **门禁**（`_verify_and_mark.mjs` 内）：① 机检无错误 ② 无灌水标记/套话 ③ 无文件内重复段。三项全过才盖章。
- **刷新表**：批量修完一段后，跑
  ```
  node scripts/_audit_all.mjs          # 重新全库审计
  node scripts/_build_fix_registry.mjs # 据审计+修复标志重建表，已修好的自动翻 ✅已修复
  ```
  重建逻辑：文件带 `<!--repaired-->` 且机检无硬错误、或已从审计问题集消失 → 状态置 `已修复`；否则仍 `待修复`。已修复行会保留在表里（带 ✅）供追溯，不会因"审计不再报它"而丢失。

## 4. 并行分派（给编排者）· 每轮最多 5 个 agent · 无需人工确认

**每一轮的循环（做完直接进下一轮，不要停下来问人）：**

```
① 取活并自动分组（组与组之间保证不重叠，可安全并行）
     node scripts/_split_batches.mjs 5 5
   → 打印 5 组，每组 5 个文件（共 25 个），每组含 文件路径 + lib + 问题摘要

② 一次派出 5 个 agent，第 N 个 agent 只拿第 N 组的清单
   · 每个 agent 的指令里必须写死：它自己【不许再派子agent】，必须亲手改
   · 每个 agent 只准碰自己那 5 个文件，不许碰别组的

③ 等 5 个 agent 全部返回后，核实真实战果（不要只信 agent 的口头汇报）：
     node scripts/_next5.mjs 1
   → 输出里的「已完成 N」就是真实盖了 ✅ 标志的数量，与上一轮对比即知本轮真实产出

④ 刷新总表（可每几轮做一次，非必须每轮）：
     node scripts/_build_fix_registry.mjs

⑤ 立刻回到 ①，开始下一轮。直到 _split_batches 输出「没有待修复的了」为止。
```

**要点：**
- **最多 5 个并行**，再多容易触发 API 限流，反而更慢。
- **不要向人确认**，一轮结束直接开下一轮，直到全部清空。
- **核实优先于信任**：agent 可能声称"已完成"但实际没落盘。唯一可信的完成凭证是文件里的 `<!--repaired-->` 标志（由 `_verify_and_mark.mjs` 通过门禁后自动盖）。用步骤③的数字对比来判断。
- **限流/中断无损**：文件要么已盖 ✅（下轮自动跳过），要么没盖（下轮自动重取）。任何时候重启都能无缝续上。
- 表已按 P0→P1→P2 排序，`_split_batches` 自然先清 P0。也可 `node scripts/_split_batches.mjs 5 5 P0` 只做 P0。
