# 经历（deeds）功能集成指导

把 **fanren-remake** 的「经历/事迹时间线」移植到 **zhushen-space**，覆盖 **NPC 栏位** 和 **主角栏位**。

本文档说明：① 在 fanren-remake 资产里到哪找原始实现；② zhushen-space 现状差距；③ 完整改造方案（含可直接粘贴的代码）；④ 构建与验证。

---

## 0. 一句话目标

把现在「一行行纯文本」的近况事迹，升级成 fanren-remake 那种**结构化时间线**——每条经历带 `时间 / 地点 / 描述` 三个字段，**NPC 和主角都能看**，AI 可用独立指令写入。

---

## 1. fanren-remake 里的「经历」长什么样（源码定位）

fanren-remake 前端是**预构建产物**（minified），但字段名/schema 在 `fanren-remake-public-assets/assets/` 下可读。关键定位：

### 1.1 数据 schema（权威来源）
文件：`assets/runtimePersistence-DI-l3NQM.js`

```js
// 单条经历对象 schema（vn）
vn = S({ time: f(), location: f(), description: f() })

// 角色传记 bio 块
bio: S({
  background: f(),          // 背景简介
  deeds: h(vn),             // ← 经历数组：Array<{time, location, description}>
  deedsText: f().optional(),// 可选的整段文字版
  innerThought: f().optional(),
  rawRelations: h(Cn),
  currentMotive: f().optional(),
  shortTermGoal: f().optional(),
  longTermGoal: f().optional(),
  worldEvolutionForce: g().optional()
})
```

> 要点：**`deeds` 是对象数组**，不是字符串。每条 = `{ time, location, description }`。
> 玩家（`kind:"player"`）与 NPC 共用同一个 `bio.deeds` 结构——所以主角天然就有经历。

### 1.2 经历怎么被读出/展示
文件：`assets/logic-er2yBlWs.js`、`assets/sendMessage-BHa8C26X.js`

```js

i.map(e => `${e.time} ${e.location}: ${e.description}`).join('\n')
```

显示格式就是 **`[时间@地点] 描述`**，最近一条置顶/置底。

### 1.3 初始化样例
文件：`assets/combat-BOdC8mFU.js` —— 新建客座角色时 `bio:{ background:'...', deeds:[], rawRelations:[] }`，即经历初始为空数组，靠后续演化追加。

## 2. zhushen-space 现状（差距分析）

| 维度 | fanren-remake | zhushen-space 现状 | 差距 |
|---|---|---|---|
| NPC 经历数据 | `bio.deeds: {time,location,description}[]` | `NpcRecord.deeds?: string`（`\n` 拼接，截最近 6 条）见 `npcStore.ts:59` | 无结构、无时间/地点 |
| NPC 经历 UI | 时间线渲染 | `NpcDetail.tsx` 的 `HistoryTab`「近况事迹」按行 `· xxx` 平铺，`NpcDetail.tsx:515` | 无字段分列 |
| 主角经历 | 玩家=角色，有 `bio.deeds` | **完全没有**。`characterStore` 只存 `skills/traits`；`CharacterPanel` 只有技能/词条两个 tab | 需新增 |
| AI 写入 | 结构化指令 | 仅登场判断 `deedsUpdates`（字符串）→ `appendDeed`，`App.tsx:974` | 需新增结构化指令 |

**结论**：要新增一个结构化 `Deed` 类型，NPC 侧升级（保留向后兼容），主角侧从零搭建，并加一条 AI 指令把两者打通。

**2b. 接口与实现**：把 `appendDeed` 升级为既能收字符串、也能收 Deed 对象；另加 `removeDeed`/`clearDeeds`：

```ts
interface NpcState {
  // ... 原有 ...
  appendDeed: (id: string, deed: string | Deed) => void;   // 签名升级
  removeDeed: (id: string, index: number) => void;          // 新增
```

```ts
appendDeed: (id, deed) =>
  set((s) => {
    const rec = s.npcs[id] ?? defaultNpcRecord(id);
    // 兼容：字符串 → 归一成 Deed（时间/地点留空，描述放原文）
    const entry: Deed = typeof deed === 'string'
      ? { time: '', location: '', description: deed, addedAt: Date.now() }
      : { ...deed, addedAt: deed.addedAt ?? Date.now() };
    const log = [...(rec.deedLog ?? []), entry].slice(-12); // 保留最近 12 条
    // 同步维护旧字符串字段，旧 UI/导出仍可用
    const legacy = log
      .map((d) => (d.time || d.location ? `[${d.time}@${d.location}] ` : '') + d.description)
      .slice(-6).join('\n');
    return { npcs: { ...s.npcs, [id]: { ...rec, deedLog: log, deeds: legacy, updatedAt: Date.now() } } };
  }),

removeDeed: (id, index) =>
  set((s) => {
    const rec = s.npcs[id];
    if (!rec?.deedLog) return s;
    const log = rec.deedLog.filter((_, i) => i !== index);
    return { npcs: { ...s.npcs, [id]: { ...rec, deedLog: log, updatedAt: Date.now() } } };
  }),

clearDeeds: (id) =>
  set((s) => {
    const rec = s.npcs[id];
    if (!rec) return s;
    return { npcs: { ...s.npcs, [id]: { ...rec, deedLog: [], deeds: '', updatedAt: Date.now() } } };
  }),
```

### 步骤 ④ 主角 store：新增 background + deedLog

主角的技能/词条在 `characterStore`，但**经历/背景没有归属**。两种放法，二选一：

- **方案 A（推荐）**：放进 `playerStore.ts`（key `drpg-player-evo`，已是主角域），新增一个 `profile` 子状态。
- 方案 B：在 `characterStore` 的 `CharacterData` 上加 `deeds: Deed[]`，主角=`B1`、NPC 也能各自存（更统一，但要同时改 NPC 侧读取来源）。

下面按 **方案 A** 实施（改动最小、与现有主角域一致）。在 `src/store/playerStore.ts` 增：

```ts

export interface PlayerProfile {
  background: string;   // 主角背景/出身
  deedLog: Deed[];      // 主角经历时间线
}

interface PlayerState {
  // ... 原有 ...
  profile: PlayerProfile;
  setBackground: (text: string) => void;
  appendPlayerDeed: (deed: string | Deed) => void;
  removePlayerDeed: (index: number) => void;
  clearPlayerDeeds: () => void;
}

// 初始 state 里：
profile: { background: '', deedLog: [] },

// actions：
setBackground: (text) =>
  set((s) => ({ profile: { ...s.profile, background: text } })),

appendPlayerDeed: (deed) =>
  set((s) => {
    const entry: Deed = typeof deed === 'string'
      ? { time: '', location: '', description: deed, addedAt: Date.now() }
      : { ...deed, addedAt: deed.addedAt ?? Date.now() };
    return { profile: { ...s.profile, deedLog: [...s.profile.deedLog, entry].slice(-20) } };
  }),

removePlayerDeed: (index) =>
  set((s) => ({ profile: { ...s.profile, deedLog: s.profile.deedLog.filter((_, i) => i !== index) } })),

clearPlayerDeeds: () =>
  set((s) => ({ profile: { ...s.profile, deedLog: [] } })),
```

## 4.5 进阶：生平压缩（深度记忆 / deep-summary）

经历越攒越多，全量塞进提示词会爆 token。fanren-remake 的解法不是「截断丢弃」，而是**滚动压缩成生平档案**——这就是「生平压缩 / 深度记忆」系统。本节说明它在哪、怎么跑，以及如何在 zhushen-space 落地。

### 4.5.1 源码定位

- **配置 + 提示词**：`assets/store-glxxBUjZ.js`（`CULTIVATION_DEEP_SUMMARY_CONFIG_V1`，narrativeMemoryStore）
- **运行期分层 + 触发 + 防过压缩**：`assets/uiStore-Bda3MKAg.js`（函数 `jL`/`ML`/`NL` 及注入逻辑）
- **持久化**：IndexedDB（`db` 模块），键名：
  - `CULTIVATION_DEEP_SUMMARY_CONFIG_V1`（配置）
  - `CULTIVATION_DEEP_SUMMARY_TEXT_${archiveId}`（当前生平正文）
  - `CULTIVATION_DEEP_SUMMARY_HISTORY_${archiveId}`（历史版本，最多 12 条）
- 生平快照挂在存档的 `lifeSnapshots[]` 上，每个快照 `{ createdAt, deepSummaryText, state{characters,…} }`。

### 4.5.2 核心机制（三层记忆 + 滚动压缩）

按「距离当前的远近」把历史分三层（`jL(idx, cfg)`，idx=从最近往前数）：

| 层级 | 条件 | 处理 |
|---|---|---|
| `full` 全文 | `idx < recentFullTextCount` | 原文保留 |
| `small` 小摘要 | `idx < recentFullTextCount + smallSummaryWindowSize` | 压成短摘要 |
| `large` 深度总结 | 更早的全部 | 并入「生平/深度记忆」长档 |

- **自动触发**（`ML`）：`enabled && autoDeepSummary && 记录条数 >= deepSummaryThreshold` 时，调 AI 把 `large` 段压进 `<deep_summary>`。
- **取用**（`NL`）：从 `lifeSnapshots` 里挑 `createdAt` 最新的 `deepSummaryText` 作为当前生平。
- **注入**（`wrapPastMemory`）：非 `full` 的记忆前缀 `[过往记忆]: `，整体包成 `<过往记忆>…</过往记忆>` 作为一条 `system` 消息插到消息列表最前。

**默认配置**（`store-glxxBUjZ.js` 的 `d`）：

```js
{
  enabled: false,           // 总开关（默认关）
  autoDeepSummary: false,   // 自动深度总结（默认关）
  deepSummaryThreshold: 100,// 累计 100 条触发
  recentFullTextCount: 5,   // 最近 5 条留全文
  smallSummaryWindowSize: 25,// 再往前 25 条做小摘要
  wrapPastMemory: true,     // 用 <过往记忆> 包裹注入
  // 历史版本上限 a = 12
}
```

### 4.5.3 关键设计：防「过度压缩」护栏

最重要的一条——**压缩不能把生平越压越短**（否则记忆会逐轮蒸发）。`uiStore` 在写回前校验新档长度：

```js
// 旧档 r 字；新档 n。允许的最小长度 i：
const i = r >= 300 ? Math.max(300, Math.floor(r * 0.3)) : 300;
if (r > 0 && n.length < i) {
  // 拒绝覆盖，理由："深度总结相对旧档压缩过度（旧档 r 字，新档 n 字），已跳过覆盖。"
  return { ok: false, reason: ... };
}
```

即：**新生平至少 300 字，且不得低于旧档的 30%**，否则放弃这次压缩、保留旧档。提示词里也有对应硬性要求（见下）。

### 4.5.4 压缩提示词（已从源码解码，含轮回乐园适配提示）

`deepSummaryPrompt`（原文为修仙语境，"每10层"需改为"每10回合/剧情小节"）：

```
你是一名专业的小说档案管理员。
请阅读下方提供的剧情记录，将其整理为清晰、可回看的深度记忆档案。
请按"约每10回合 或 剧情小节"为单位进行分段记录，并在整合时保留已有深度记忆中的关键事实。

硬性要求：
1. 最终 <deep_summary> 正文至少 300 字，推荐 300-500 字；已有档案本来就更长时，允许继续超过 500 字。
2. 绝对不能把已有深度记忆压缩成一句话、单段描述或几条极短 bullet。
3. 必须保留关键阶段、重要人物关系变化、未完任务、当前局势，不得因为"精炼"而删空旧事实。
4. 如果新增内容较少，也要在旧档案基础上做增量更新，而不是回缩成更短版本。

请严格遵守以下输出格式（保留XML标签）：

<deep_summary>
<过往事件(可按时间段或剧情段整理)>
{剧情小节}: [时间/地点]
剧情概括: (忠实记录NPC与主角的言行举止，主角经历的事件，保留关键细节)
[重要NPC记录]
- NPC名字: 共同经历事件 | 态度转变 | 当前关系 | 当前进行的事件
- NPC名字: ...

(根据输入内容的长度，请重复上述结构及其详细信息，确保不遗漏)
</deep_summary>

{{content}}
```

> `{{content}}` 处替换为待压缩的旧经历 + 已有生平档（增量压缩）。

### 4.5.5 在 zhushen-space 落地（按需）

zhushen-space 现在是「硬截断」：NPC `deedLog` slice 最近 12、主角 20（见步骤 ②④）。要换成生平压缩，做法：

1. **数据**：给主角 `playerStore.profile` 和 `NpcRecord` 各加 `lifeSummary: string`（生平正文）+ `lifeSummaryHistory?: {text,createdAt}[]`（可选历史，上限 12）。
2. **分层**：`appendDeed` 不再粗暴 slice。改为：`deedLog` 只留最近 `recentFullTextCount`(5) 条全文；溢出的旧条目排队进「待压缩区」。
3. **触发**：当 `deedLog` 累计达阈值（如 30/100），把待压缩区 + 现有 `lifeSummary` 一起喂给上面的 `deepSummaryPrompt`，调一次 AI（可复用主角/NPC 演化的 API），产出新 `lifeSummary`。
4. **护栏**：写回前套用 4.5.3 的长度校验——`新档 < max(300, 旧档*0.3)` 则拒绝覆盖、保留旧档并 `console.warn`。
5. **注入**：构造发给主 API 的消息时，把 `lifeSummary` 以 `<过往记忆>…</过往记忆>` 形式作为 system 注入（对应现有 `App.tsx` 拼 system prompt 的位置）。
6. **持久化**：`lifeSummary`/`lifeSummaryHistory` 随 `drpg-player-evo` / `drpg-npc` 的 `persist` 一起落 localStorage，无需 IndexedDB。
7. **UI**：在「经历」tab（步骤 ③⑤ 的 `HistoryTab`/`PlayerHistoryTab`）顶部加一块「📜 生平」，展示 `lifeSummary`（只读 + 「重新压缩」按钮手动触发）。

> 最小可用版：先只做「主角生平」一条线（步骤 1/3/4/7），NPC 暂沿用截断，验证压缩质量与护栏后再推广到 NPC。
> 配置项（阈值/全文条数/窗口）建议放进 `settingsStore` 或 `playerStore`，给用户一个开关——和 fanren-remake 默认关闭一致，避免无谓的额外 API 调用。

### 4.5.6 实际实现（已落地，2026-06）

本仓库已按上述思路实现了一版**记忆短/长期模型**的生平压缩（比 §4.5.5 的 lifeSummary 草案更接近原版）：

- **数据**：工作记忆存在 `characterStore` 的 `CharacterData.memory = { shortTerm: MemoryEntry[]; longTerm: MemoryEntry[] }`（主角 `B1`、NPC `Cx` 统一），`MemoryEntry = {time, location, content}`。新增 action `appendMemory` / `setMemory`。
- **写入**：AI 指令 `addMemory("B1"/"C1", {time,location,content})` → 追加到 `shortTerm`（在 `stateParser.ts` 与 `addDeed` 同体系解析）。
- **配置**：新 store `memoryStore.ts`（key `drpg-memory`）持有开关、范围(both/player/npc)、阈值/保留数、**轮回乐园适配版压缩提示词** `DEFAULT_MEMORY_PROMPT`，以及**独立 API 配置**（`memoryApi`/`memoryUseSharedApi`/`fetchMemoryModels`，可共用正文 API）。
- **压缩**：`App.tsx` 的 `runMemoryCompressionPhase()` 接入 `runPostNarrativePhases`，并发不阻塞；按范围筛出 `shortTerm≥阈值 || longTerm≥阈值` 的角色，**批量**喂给提示词（`${characters_payload}`），用生平压缩自己的 API；解析返回 JSON `{results:{id:{shortTerm,longTerm,bio}}}`，按保留数截断后 `setMemory`，`bio` 回写主角 background / NPC 第10列。
- **UI**：「经历」tab 内新增只读「记忆」区（近期/长期）；**设置 → 变量管理 → 「📜 生平压缩」按钮** → 独立 `MemoryManager`（页面 `memory-manager`），含「压缩设置」（开关/范围/阈值/提示词直接编辑+恢复默认）与「API 设置」两个 tab。
- **预设**：`addMemory` 已加入两个预设的指令白名单与说明。
- **与 deeds 的关系**：`deedLog`（经历时间线）是给玩家看的事件流，**不被压缩**；`memory`（short/long）才是会被 AI 压缩整理的认知记忆。两者并存、互不影响。

> 默认 `enabled:false`（与原版一致，避免额外 API 调用）；在该 tab 打开后生效。

## 6. 向后兼容与注意事项

- **不要删** `NpcRecord.deeds`（字符串）字段：旧存档、登场判断 `deedsUpdates`、其他导出仍依赖它。新逻辑同时维护新旧两份。
