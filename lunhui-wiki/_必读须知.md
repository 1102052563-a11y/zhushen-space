5. **一个事实一个正主**：NPC 详情写在该 NPC 页，其它页只**链接**、不复制。每个人物页含「**关系**」一节（列其与他人关系、互链）。人物**死亡 / 状态变化时更新 front-matter 的 `状态` 字段**（如「已死亡（二卷64章被斩杀）」）。
6. **格式**：内部链接用相对 `.md`；提示框用 `!!! tip "标题"`；**新建页必须加进 `mkdocs.yml` 的 nav**。
7. **主角 / 常驻角色**：当前值**覆盖**到最新 + 成长轨迹**只增不删**；区分**持久**（跨世界）/ **世界限定**两层；角色页=纵向（当前状态/成长轨迹/历程按世界块），任务世界页=横向（角色在本世界 + 世界结束存**结算快照**）。结算快照含：① 世界结算（世界之源/任务数/评价/奖励）；② **离世定格**——等级/职业/天赋/技能/六维/法力/币/属性点 + **当前装备（武器·防具·饰品，逐件）** + 携出物 / 留下物，须注明「此为离世瞬间、与当前状态不同」；③ **随从/召唤物快照**——苏晓名下若有随从/契约生物/召唤物，逐个记离世定格（名称/来源/属性/能力/去向：带出·留下·阵亡·解约）。**苏晓的随从/召唤物每个各单开一页**（放 `docs/人物/`·头号随从=`布布汪.md`·nav「主角与随从」分组下逐条列·新随从在布布汪下加条目·各世界快照写在该随从页·与苏晓主页分开），世界页只摘要+链接。
10. **排版铁则（防 list 渲染崩坏）**：Markdown 列表**前面若有正文段落/介绍句，二者之间必须空一行**——否则 python-markdown 会把 `- ` 当成段落里的字面文字、整块挤成一坨（已踩坑：势力页斩赤段）。规律：`## 标题`直接接`- 列表`没问题；但`段落句。`紧跟`- 列表`必崩，须空行。改完可用扫描脚本自检（见 PROGRESS）。
8. **分类演进**：大类按需加（**加法优先、不破坏**已有路径），条目 / 子页自动加；一时归不进的先丢「待归类」。大类调整先说一句理由再动。
9. **新机制 / 新大条目要立条目**：原文一出现够分量的新事物——**新机制**（如阵营任务、命运之力、声望、临时契约）、新势力 / 新生物 / 新地点 / 新道具或材料类别、重要新人物——就**新建专门页或在对应大类页新增小节**（标章节出处、加 nav），别让它淹没在某段叙述里；小角色 / 一次性物件可先并入对应 index。判断够不够分量：是否会被反复引用、是否影响力量 / 世界观体系。

## §2 每批流程
读 N 章 → 抽实体（人物 / 阶位天赋技能属性 / 装备道具消耗品 / 势力 / 任务世界 / 生物万族 / 地点 / 事件）→ 增量建或改 `.md`（+ 改 nav）→ `时间线.md` / `章节梗概` 追加 → **`mkdocs build --strict` 验证零死链** → 更新本文件 §0 + `PROGRESS.md` 变更记录。

## §3 命令
- 验证（死链即报错）：`python -m mkdocs build --strict -f lunhui-wiki/mkdocs.yml`
    - ⚠️ 输出里的「**Warning from the Material for MkDocs team**」是无害推广信息、**不是错误**；只看含 `WARNING` / `ERROR` 且与链接 / nav 有关的行；见 `Documentation built in …` 即成功。
- 预览（热重载，端口 5180）：`python -m mkdocs serve -f lunhui-wiki/mkdocs.yml -a 127.0.0.1:5180`（端口常已被占用＝服务在跑，直接开浏览器即可）。
