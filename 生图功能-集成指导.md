## 0. 总览

三条相互独立的生成线，各自可选服务商，各自有提示词模板与触发开关：

| 生成线 | 产物落点（zhushen 已就绪字段） | 触发 |
|---|---|---|
| **肖像生成** | `NpcRecord.avatar`（NPC）/ 主角立绘（新增 `PlayerProfile.avatar`）| 自动（无图在场角色）/ 手动（浮窗·详情上传旁「AI 生成」）|
| **装备生图** | `InventoryItem.image`（主角）/ `NpcOwnedItem.image`（NPC，需新增）| 自动（有外观无图的装备）/ 手动（物品详情）|
| **正文配图** | **新增**：挂到对应 assistant 楼层的 `images[]`（每张含 anchor/url/prompt/nsfw）| 自动（每回合抽 N 锚点）/ 手动（右键回复）|

服务商：**NovelAI(NAI) / OpenAI(图片) / Gemini(图片) / ComfyUI / 自定义(OpenAI 兼容)**。自定义按 肖像/正文/装备 拆三套互不影响。

> **已有基础**：`NpcRecord.avatar` 和 `InventoryItem.image` 字段、右上角在场人物浮窗、左上角主角装备浮窗、详情页上传/换图按钮都已实现——生图只需把生成结果写进这些字段即可显示。正文配图是全新的展示位。

## 1. 服务商 API 形态（照搬原版请求体）

### 1.1 NovelAI / NAI
- **URL**：用户填 `https://image.novelai.net`，程序自动补 `/ai/generate-image`（原版 `Un()`：末尾无 `/ai/generate-image` 则补全）。
- **鉴权**：`Authorization: Bearer <Persistent API Token>`。
- **请求体**（原版 `$n()`）：
  ```jsonc
  {
    "input": "<正向提示词：画师串 + 角色tags + 正向后缀，逗号连接>",
    "model": "nai-diffusion-4-5-full",
    "action": "generate",
    "parameters": {
      "params_version": 3,
      "width": 1024, "height": 1024,
      "steps": 28,
      "scale": 5,                       // = Prompt Guidance(0~10)
      "sampler": "k_dpmpp_2m_sde",      // UI「DPM++ 2M SDE」→ k_dpmpp_2m_sde
      "n_samples": 1, "ucPreset": 0, "qualityToggle": true,
      "sm": false, "sm_dyn": false, "dynamic_thresholding": false,
      "controlnet_strength": 1, "legacy": false, "add_original_image": false,
      "legacy_v3_extend": false,
      "noise_schedule": "karras",
      "cfg_rescale": 0,                 // Prompt Guidance Rescale(0~1)
      "uncond_scale": 1,                // 负面提示词强度(0~1.5)
      // —— nai-diffusion-4* 用 v4 结构 ——
      "v4_prompt": { "use_coords": false, "use_order": false, "caption": { "base_caption": "<正向>", "char_captions": [] } },
      "v4_negative_prompt": { "use_coords": false, "use_order": false, "caption": { "base_caption": "<负向>", "char_captions": [] } },
      // —— 旧模型(nai-diffusion-3 等)用 ——
      "negative_prompt": "<负向>",
      "seed": 0                         // 可选
    }
  }
  ```
  - **模型判定**：`/^nai-diffusion-4(?:-|$)/i` → 走 `v4_prompt`/`v4_negative_prompt`；否则走 `negative_prompt`。
  - `sampler==='k_euler_ancestral'` 时额外 `deliberate_euler_ancestral_bug:false, prefer_brownian:true`。
- **响应**：NAI 返回 **ZIP 二进制包**（不是直接图片）。原版 `Jn()`：`compression===8` 时用 `DecompressionStream('deflate-raw')` 解压，取出第一张 PNG → 转 dataURL/blob。**这是 NAI 接入最容易踩的坑**，必须解 zip。
- **画师串(artistTags)**：带冒号权重（`15::best quality::, 1.2::8k::, …`），追加到正向提示词**末尾**。
- **NAI 请求队列**：开启后所有 NAI 请求**串行**，相邻请求至少间隔 N 秒（默认 10），避免同浏览器多标签并发打到 NovelAI。
- **RPM 上限**：每分钟最多 N 次，超额先尝试备用路由（OpenAI 才有路由，NAI 单接口）。

### 1.2 OpenAI / Gemini 图片（OpenAI 兼容）
- **URL**：`<baseUrl>/images/generations`。
- **请求体**（原版）：
  ```jsonc
  { "model": "gpt-image-2", "prompt": "<英文提示词>", "size": "1024x1024",
    "quality": "high", "background": "auto", "output_format": "jpeg", "moderation": "<low/auto>" }
  ```
- **主备路由 fallback**：肖像/正文各自配一列有序路由（路由1主、路由2备…），首选失败按序重试下一条——**完全复用 zhushen 已有的 `apiRoutes` + `apiChatFallback` 思路**（见 CLAUDE.md「中心 API 接口库 + 多接口路由」），新增 `image:portrait` / `image:story` 两个 featureKey。
- **参考图**：OpenAI 模式可附「在场角色肖像 / 上一回合正文图」作参考（多模态 image 输入），保持人物/场景连贯；NAI 模式不发参考图、只用 LLM 锚点+tags。

### 1.3 ComfyUI（工作流模板驱动，原版 `Vr/Fr/Br/Lr` 等）
ComfyUI 不是单一 endpoint，而是「提交工作流图 → 排队 → 轮询历史 → 取图」四步：
- **配置**：`apiUrl`（ComfyUI 地址）+ **`workflowJson`（API 格式工作流 JSON，用户从 ComfyUI「保存(API格式)」导出粘贴）** + `positiveNode`/`positiveInput`(默认 `text`) + `negativeNode`/`negativeInput`(默认 `text`) + `pollIntervalMs`(默认 1200，250~10000) + 超时。
- **注入提示词**（原版 `Fr()`）：把工作流 JSON 里指定节点的输入替换——正向→`positiveNode.inputs[positiveInput]`、负向→`negativeNode.inputs[negativeInput]`、**seed 自动探测**所有含 `seed/noise_seed` 的节点并填随机或指定种子（`Mr()`）。正向提示词同样在末尾追加画师串。
- **提交**（`Vr()`）：`POST {apiUrl}/prompt`，body `{ "prompt": <注入后的工作流>, "client_id": "<固定id>" }` → 返回 `prompt_id`。
- **轮询**（`Br()/zr()`）：`GET {apiUrl}/history/{prompt_id}`，每 `pollIntervalMs` 查一次，直到 `outputs.*.images[]` 出现首图；若 `status.completed && status_str!=='success'` → 抛错；超时报"工作流可能仍在运行"。
- **取图**（`Lr()`）：`GET {apiUrl}/view?filename=&subfolder=&type=` 拿到图片二进制 → dataURL。
- ComfyUI 适合本地/私有部署、完全自定义工作流；缺点是要用户自己配工作流 JSON 和节点 id。

## 3. 生成器 `systems/imageGen.ts`

统一出入口，屏蔽服务商差异：

```ts
// 返回 dataURL（或 blob url）
export async function generateImage(opts: {
  service: 'nai'|'openai'|'gemini'|'comfy'|'custom';
  prompt: string; negative?: string; size?: string;
  refImages?: string[];          // OpenAI 参考图
  signal?: AbortSignal;
}): Promise<string>;
```
- `nai`：拼 body（§1.1）→ fetch → **解 ZIP**（`DecompressionStream('deflate-raw')`，含不支持时报错）→ 取首图 → dataURL。
- `openai`/`gemini`/`custom`：`resolveApiChain('image:xxx', legacy)` → 逐路由 fetch `/images/generations`（§1.2）→ 取 `data[0].b64_json`/url → dataURL。失败 fallback 下一路由。
- `comfy`：注入工作流（正/负/seed 节点）→ `POST /prompt` 拿 `prompt_id` → 轮询 `/history/{id}` 直到出图 → `/view` 取图 → dataURL（§1.3）。
- **队列**：NAI 走全局串行队列（`queueEnabled` 时所有 NAI 任务进队，间隔 `queueGapSec`）。
- **超时**：`AbortController` + `timeoutSec`（留空不超时）。

## 4. 三条流水线

### 4.1 肖像
1. 选目标：自动=在场且无 avatar 的 NPC（+ 主角无立绘时）；手动=指定角色。
2. 拼提示词：优先用角色 `imageTags`（NAI tags / col19）；否则按档案（外观描写/性别/年龄/阶位/职业/标签）+ `portraitPromptFormat` 拼。NAI 末尾追加画师串。
3. `generateImage(portraitService, …)` → 写 `NpcRecord.avatar` / 主角 avatar。
4. 开关：`autoSettingSheet`（先设定图后主肖像）、`concurrentPortrait`（并发/串行）、`refreshOnLook`（角色演化写入"外观变化标记"后自动重绘——需重点演化输出一个 flag）。

### 4.2 装备
1. 选目标：自动=有 `appearance` 且无 `image` 的装备（玩家/NPC）；手动=物品详情。
2. 用 `equipPromptTemplate` 替换 `${item_name}/${item_category}/${item_grade}/${item_description}/${item_appearance}/${item_effect}/${owner_gender}/${wearer_gender}/${portrait_style}` → 提示词（**单件装备设定图**，无人物/手/水印/现代元素，见用户给的装备模板全文）。
3. `generateImage(equipUsePortraitService ? portraitService : equipService, …)` → 写 `InventoryItem.image`/`NpcOwnedItem.image`。

### 4.3 正文配图（最复杂）
1. **独立 LLM 路由**（`storyLlmRoutes`，与重点演化路由分开）跑 `storyPromptTemplate`：输入本回合正文 + 在场角色完整外观 + 场景/时间/新登场角色 + `image_count`，输出 `image_count` 个 `<image>` 块，每块含：
   - `<anchor>`：**逐字复制正文中的连续短片段（8~30字，可 Ctrl+F 命中）**——用于把图插到正文对应位置。
   - `<nsfw_rating>`：`sfw`/`nsfw_mild`/`nsfw_moderate`/`nsfw_explicit`（按画面实际呈现判定，分级规则见模板）。
   - `<prompt>`：英文 NAI tags，**主体数量+性别开头**（`1girl`/`1boy`…），不含 col19/中文/负面/Markdown。
2. 逐 `<image>`：`prompt` + 画师串 + (按 nsfw 等级追加的裸露/行为 tags) → `generateImage(storyService, …)`（NAI 逐张独立；OpenAI 可发参考图）。
3. 把生成图按 `<anchor>` 在该 assistant 楼层正文里**定位插入**（找到 anchor 子串的位置，在其后插入 `<img>`），存进 `message.images[]`。
4. 开关：`autoStoryImage`、`storyImageCount`、参考图三连（`storySendReference`/`storyRefOnsceneAvatars`/`storyRefPrevImages`）、`storySendFullText`（NAI 固定 false）。

## NAI（及 OpenAI/Gemini 官方端点）CORS 问题与代理

**现象**：点「✨AI 生成」报 `Failed to fetch`。这是浏览器层面请求没送达（不是接口报错码）。**根因**：NAI 官方端点 `https://image.novelai.net/ai/generate-image` 不返回 CORS 头，浏览器跨域被拦。fanren 之所以能用，是因为它**不直接调 NAI**——而是请求自己部署的代理（`Wt()`：本地地址直连，其余走代理 `A`，用 `X-Upstream` 头带真实地址转发并补 CORS 头）。

**zhushen 的实现**：`NaiConfig.corsProxy` + `imageGen.applyProxy(proxy, realUrl)`：
- `corsProxy` 含 `{url}` → 前缀式：`fetch(proxy.replace('{url}', encodeURIComponent(realUrl)))`。
- 否则 → 头式（兼容 fanren）：`fetch(proxy, { headers:{ 'X-Upstream': realUrl, ... } })`。
- 留空 = 直连 = 必然 `Failed to fetch`。UI 在「生图API配置 → NAI」有「CORS 代理地址」字段。

**可一键部署的 Cloudflare Worker（两种写法都兼容）**：
```js
export default {
  async fetch(request) {
    const cors = {
      'Access-Control-Allow-Origin': '*',
      'Access-Control-Allow-Methods': 'GET,POST,OPTIONS',
      'Access-Control-Allow-Headers': 'Content-Type, Authorization, X-Upstream, Accept',
    };
    if (request.method === 'OPTIONS') return new Response(null, { headers: cors });
    let upstream = request.headers.get('X-Upstream');
    if (!upstream) upstream = new URL(request.url).searchParams.get('url');
    if (!upstream) return new Response('missing upstream', { status: 400, headers: cors });
    const fwd = new Headers(request.headers);
    fwd.delete('X-Upstream'); fwd.delete('host');
    const resp = await fetch(upstream, {
      method: request.method, headers: fwd,
      body: (request.method === 'GET' || request.method === 'HEAD') ? undefined : request.body,
    });
    const out = new Headers(resp.headers);
    for (const [k, v] of Object.entries(cors)) out.set(k, v);
    return new Response(resp.body, { status: resp.status, headers: out });
  }
};
```
部署后把 Worker 域名填进「CORS 代理地址」（头式直接填 `https://xxx.workers.dev`；或前缀式填 `https://xxx.workers.dev/?url={url}`）。该 Worker 会透传 Authorization 与二进制 ZIP 响应，NAI 解 ZIP 逻辑照常工作。

## NAI 解 ZIP 实战要点（`imageGen.extractImageFromZip`，最终方案=移植 fanren `Yn`/`Jn`）
NAI `/ai/generate-image` 返回 **ZIP 二进制包**（`Content-Type: binary/octet-stream`，本地头 flag `08 00` = 带**数据描述符**，本地头里的压缩大小写成 0），里面单个 PNG 多为 **deflate 压缩**。
**最初用"扫 PNG 头 + 从尾找 IEND + deflate 兜底"的启发式 → 遇数据描述符切错 → 反复"格式不对"。最终改成 fanren 的正规 ZIP 解析（最稳）：**
1. 首字节 `89504E47`/`FFD8`/`52494646` → 已是 PNG/JPEG/WebP 原图，直接用；`50 4B`(PK)→ZIP 继续。
2. **优先读中央目录**（签名 `0x02014b50`）：从目录条目读 **压缩方式(off+10)/压缩大小(off+20)/文件名长(off+28)/extra(off+30)/comment(off+32)/本地头偏移(off+42)**——**中央目录的尺寸永远准确**（不受数据描述符影响），这是关键。跳到本地头(`0x04034b50`)按其 nameLen/extraLen 算出数据起点，按文件名后缀(`.png/.jpg/.webp`)挑图。
3. 退化：扫本地文件头(`0x04034b50`)。
4. 按压缩方式还原：`method 0`=stored 直接用 / `method 8`=`DecompressionStream('deflate-raw')` 解压（`inflateEntry`）。mime 由首字节嗅探。
> 调试：console fetch 取 `arrayBuffer()` 看前 8 字节判 ZIP/原图；`blob.size` 只证明字节回来了、**不能证明解码成功**，务必 `URL.createObjectURL` 实际预览。

## fanren vs zhushen 生图差异（对比确认）
- **解 ZIP**：fanren 用正规中央目录解析（`Yn`+`Jn`），zhushen 已移植同款（见上）。
- **缩图/重编码**：**fanren 不缩图**、直接存原图；它把生成图落 **IndexedDB / 后端云存档**（"后台生图落库"），不进 localStorage，故无配额问题。**zhushen 存 localStorage(5MB)**，所以加了 `shrinkDataUrl`（canvas 重编码小 JPEG，头像512/装备384）来适配——这是架构差异，非 fanren 做法。若以后把头像/装备图也迁到 IndexedDB，可去掉缩图存原图。
- **代理**：fanren 走自建代理(X-Upstream)，zhushen 走用户填的 `corsProxy`(兼容同款头式)。

## 头像/装备图体积（localStorage 配额）
生成的图 dataURL 约 1~3MB，直接写 `PlayerProfile.avatar`/`NpcRecord.avatar`/`InventoryItem.image`/`NpcOwnedItem.image`（都在 localStorage）会很快 **`exceeded the quota`**。解决：`imageGen.shrinkDataUrl(dataUrl, maxDim=768, q=0.82)`——canvas 缩到最长边 ≤768、转 JPEG（→ ~50-120KB），在**所有头像/装备的生成与上传写入点**调用（PlayerSidebar/NpcDetail AvatarBlock+NpcItemCard/BackpackModal ItemImageBlock/App runPortraitPhase+runEquipImagePhase）。**正文配图存 chatDb(IndexedDB) 不缩**。

## 注意事项
